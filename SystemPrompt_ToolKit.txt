## 第一部分：System Prompt 应该怎么写？

System Prompt 不负责教 AI 具体怎么用工具，而是负责塑造 AI 的“人格”、规范其沟通方式，并划定不可逾越的底线。

### 1. 角色定义与专业客观性 (Role & Objectivity)
* **明确设定身份**：开篇直接告诉它是什么（例如：“你是一个交互式 CLI 工具，负责协助用户完成软件工程任务”）。
* **拒绝无意义的附和**：明确要求 AI 把技术准确性放在首位。告诉它：“不要为了迎合用户而提供虚假的情绪验证。专注于事实和解决问题，遇到错误想法时要客观地纠正。”
* **避免说教**：如果因为安全策略无法提供帮助，要求它直接拒绝并提供替代方案，绝对不要长篇大论地解释“为什么这很危险”，这会显得非常烦人。

### 2. 主动性与边界管理 (Proactiveness vs. Boundaries)
* **不要给用户“惊吓”**：AI 可以主动，但不能越界。明确规定：“绝不主动创建 README 或文档，除非用户明确要求”；“绝不主动 Commit 代码，除非用户下达了明确指令”。
* **安全红线 (Guardrails)**：清晰列出禁止事项。对于编程 Agent，必须强调“仅协助防御性安全任务，拒绝编写恶意代码，拒绝协助窃取凭据（如批量扫描 SSH 密钥）”。

### 3. 动态上下文注入 (Dynamic Context)
预留特定的 XML 标签（如 `<env>`），在每次 API 请求前动态注入当前系统环境。
* **包含要素**：当前操作系统版本、工作目录（绝对路径）、是否有 Git 仓库、以及**今天的具体日期**（防止 AI 在搜索或写代码时用错时间）。

## 第二部分：Tools Prompt 应该怎么写？

Tools Prompt 通常以列表形式追加在 System Prompt 之后。它不仅要提供 JSON Schema 供接口调用，更要提供详尽的**自然语言操作手册**。

### 1. 明确工具的“不适用场景” (Anti-patterns)
描述一个工具时，不仅要说它能干嘛，更要**强调它不能干嘛**。
* **例如**：在定义 Bash 工具时，必须严厉声明：“绝不要用 Bash 里的 `cat`、`sed`、`grep` 来读写和搜索文件！必须使用专门的 Read、Edit 和 Grep 工具！” 这种指令能有效防止 AI 绕过专门优化的工具去执行低效的系统命令。

### 2. 建立严格的操作协议 (Strict Protocols)
对于破坏性较高的工具（如终端执行、Git 操作），必须在文本描述中写明 Standard Operating Procedure (SOP)。
* **环境安全协议**：例如规定“在使用命令创建新目录前，必须先 `ls` 验证父目录是否存在”。
* **Git 隔离协议**：明确警告“绝不运行 `push --force`”、“绝不跳过 Git Hooks”、“修改 Commit 前必须先检查作者信息”。

### 3. 强调性能优化与并行调用 (Batching & Parallelism)
* 明确告诉大模型：**你具备在一次回复中调用多个工具的能力**。
* 给出具体场景指导：如果几个命令互不依赖（比如 `git status` 和 `git diff`），必须将它们打包在一个消息里并行发出，以极大地提升执行效率和用户体验。

### 4. 提供复杂的 Few-Shot 语法示例
如果工具需要特定格式的参数，直接在描述里用 `<example>` 演示。
* **例如**：如果需要 AI 提交一段包含多行文本的 Pull Request，告诉它必须使用 Bash 的 HEREDOC 语法（`cat << 'EOF'`），以防止引号和转义符破坏命令格式。

## 第三部分：核心基础工具箱 (The Standard Toolkit)

在不包含专门的“任务规划器（如 TodoWrite）”和“多代理（Sub-agents）”的情况下，一个健壮的 AI 助手至少需要以下三组核心工具：

### 组别一：终端与进程控制 (Bash Execution Group)
* **`Bash` (主终端执行器)**
  * **用途**：执行实际的系统命令（如 `npm run dev`, `docker build`, `git pull` 等）。
  * **设计要点**：必须提供 `timeout`（超时时间，防止进程卡死）和 `run_in_background`（后台运行，用于启动服务）参数。强烈要求 AI 对含有空格的路径使用双引号。
* **`BashOutput` (后台输出读取器)**
  * **用途**：当 Bash 命令被设为后台运行后，使用此工具通过 `shell_id` 抓取该进程产生的新日志（stdout/stderr）。
  * **设计要点**：支持可选的 `filter` 正则参数，以便在海量日志中只抓取关键报错。
* **`KillShell` (进程终结器)**
  * **用途**：通过 `shell_id` 强制终止一个正在后台运行的、可能失控的 Bash 会话。

### 组别二：精准文件读写 (File I/O Group)
* **`Read` (文件读取器)**
  * **用途**：直接读取本地文件、图片或 PDF。
  * **设计要点**：必须支持 `offset`（起始行）和 `limit`（读取行数），以防止读取巨型日志文件时撑爆 LLM 的上下文窗口。
* **`Write` (文件写入/覆盖器)**
  * **用途**：将全新内容写入绝对路径。
  * **设计要点**：强制要求在覆盖已存在文件前，AI 必须先调用 `Read` 读取原文件（可以在工具逻辑层面对此做硬性阻挡）。
* **`Edit` (精确内容修改器)**
  * **用途**：在不重写整个文件的情况下修改代码。
  * **设计要点**：接受 `old_string` 和 `new_string`。必须警告 AI：旧字符串必须完全匹配（包括空格和缩进），且在文件中必须唯一，否则替换会失败。

### 组别三：高效代码检索 (Search & Retrieve Group)
* **`Glob` (路径模式匹配)**
  * **用途**：极速寻找特定名称或后缀的文件（如 `**/*.tsx`）。替代低效的 Bash `find`。
* **`Grep` (内容正则检索)**
  * **用途**：基于底层的 ripgrep，在庞大的代码库中进行正则表达式搜索。
  * **设计要点**：支持返回“仅包含匹配项的文件列表”或“具体匹配的上下文（前后N行）”，支持跨行匹配开启（multiline）。

### 组别四：外部信息获取 (Web Access Group)
* **`WebFetch`**：输入 URL，抓取网页并自动将 HTML 转换为易读的 Markdown。
* **`WebSearch`**：进行互联网检索，用于补充大模型知识截止日期之后的最新技术文档或新闻。
